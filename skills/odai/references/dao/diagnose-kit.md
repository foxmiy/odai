# Diagnose Kit

## 何时读取

诊断是 `references/dao/inquiry-discipline.md`（通用查证止损纪律）的特化：进 bug / 报错 / 异常 / 行为不符 / 性能回归的诊断（主阶段 = 诊断）时，先按 inquiry-discipline 的判别门 / 止损 / 降格 / 假设纪律，再读本 kit 补 bug 诊断特有的红信号闭环那套。判入来源：`道`「何时必须读取 support files」直接判入，或 `implement-code` 返工止损（规则 17）判定绕圈后转入。本文件只补“硬 bug 诊断纪律”；证据账本与诊断起草结构（已确认现象、疑似原因、已有证据、缺失证据、最有价值排查动作）以 `references/dao/review-kit.md` 为准，红-绿-重构与回归测试以 `references/modules/implement-code.md` 与 `references/implement-code/tdd-kit.md` 为准，不重复。

诊断属法层先手：先定“如何验证、何时止损”，再谈根因。下面这条铁律是法层先手在调试场景的具体化。

## 铁律：先建红信号闭环，再谈假设

建出一个能对准症状变红的反馈闭环，bug 就已解决九成；二分、假设、探针都只是消费这个闭环。没有闭环，盯着代码看再久也补不回来。把不成比例的力气花在建闭环上：要主动、要会变招、不轻易认输。

**fail-closed**：拿不出一条已实际跑过的红信号命令，就停在建闭环，不许进入假设。一旦发现自己“还没有这条命令就开始读代码编理论”，立即停——这正是本 kit 要拦的失败。

“停在建闭环”不是 blocked、更不是无事可做：只有症状、缺复现输入 / 环境 / 权限时，**建闭环本身就包含**只读补证、向用户要复现条件与运行环境、设计最小红信号命令、把偶发复现率抬到可调试。这些都在推进闭环、是当前最高价值动作；fail-closed 只拦“无闭环就编根因假设”，绝不拦“为建闭环而补证、提问与试构造”。只有补证与提问都做了仍无任何可跑路径（缺凭证 / 缺外部状态）时，才按契约 blocked 分型提问。

## 止损重估（诊断特化）

通用的判别门、止损（换方向不换变体）、降格规则、假设纪律以 `references/dao/inquiry-discipline.md` 为准；诊断里它们的信号与动作特化如下：

- 未收敛的信号 = **闭环两轮未变绿**（同一验收口径连续两轮未达成，无论失败形态是否相同）。
- 更强的止损信号 = **假绿**：改动已自称变绿或完成，但用户实测反馈仍不对——直接证明闭环准入①“断言用户确切症状”被违反，测的不是用户实际在做的事，不能拿“我当场推断应该绿了”顶替。
- 降格规则里“唯一不降的观察事实”在诊断中就是**用户观察到的现象**（具体操作 / 输入 → 具体错误结果），它即闭环要对准复现的靶子；“坏在哪 / 为什么坏 / 怎么改”全是待证伪假设。
- 止损动作除“换方向不换变体”外，补两条 bug 专属：① 闭环优先于再改一版——连续改不好几乎总是红信号闭环没建对或压根没建（fail-closed 被绕过），先回「闭环准入 4/4」补闭环，而不是盲改第三版；② 假绿以用户实测为准重建闭环——权威症状是用户的真实复现，闭环必须重建到能复现它、而非自己那条自称绿的路径，先回读本会话每轮改动与用户每轮反馈（配合「对照与新近改动」，命中时叠加记忆里本仓“假绿陷阱 / 验证盲区”的稳定事实）定位“自称绿”与“实际行为”的差，不重启猜测。

## 闭环准入（可检查判据，外显一行）

能指名一条**已实际跑过至少一次**的命令（贴出调用与它的输出），且四条全中：

1. 能变红：真正打到 bug 代码路径，断言**用户描述的确切症状**，能因此 bug 变红、修好后变绿；不是“跑通不报错”。
2. 确定性：同输入同判定；偶发 bug 指复现率已抬到可调试。
3. 快：秒级，不是分钟级。
4. 可无人值守跑：需要人点的，只能经一次性 HITL 脚本驱动，把人也纳入闭环。

四条全中才算闭环已建，外显「诊断闭环 4/4」；写不出命令或缺任一条，按 fail-closed 停在建闭环，不进入假设。

## 闭环构造菜单（大致按此序试）

失败测试（单元 / 集成 / e2e）→ curl / HTTP 脚本打开发服务 → CLI + 夹具输入、stdout diff 已知正确快照 → 无头浏览器脚本断言 DOM / console / network → 回放抓取的真实 trace / payload → 一次性 harness 只起最小子系统跑这条路径 → 属性 / fuzz 跑千次随机输入找失败模式 → 二分 harness 配 `git bisect run` → 差分新旧版本或配置、diff 输出 → 人在环 bash 脚本（最后手段）。

## 收紧闭环

有了闭环就当产品打磨：更快（缓存初始化、跳无关 init、缩测试范围）、信号更尖（断言具体症状而非“没崩”）、更确定（钉时间、播种随机、隔离文件系统与网络）。30 秒的偶发闭环约等于没有；2 秒确定性闭环才是调试超能力。偶发 bug 的目标不是干净复现，而是把复现率抬到可调试：循环触发、并行、加压、缩时窗、注 sleep——50% 可调，1% 不可调，先把率抬上去。

## 最小化复现

变红后，把场景砍到“再砍一条就变绿”的最小集：逐条删输入、调用方、配置、数据、步骤，每删一条重跑闭环，只留承重的。最小集既缩小假设空间，又直接当回归测试种子（回归测试落 `implement-code`）。

## 对照与新近改动（缩小假设空间的廉价首动作）

建闭环或排假设时，两个廉价动作优先做，常一步把假设空间砍掉大半：① 先看**新近改动**——`git diff`、最近 commit、依赖与配置变更；回归类 bug 多半就在这里，先验它能极快缩小范围。② 找同仓一个**确实能工作**的相似实现，完整读（不选择性扫几行），逐条列出它与坏路径的**每一处差异**——依赖、配置、调用顺序、环境假设、入参形状；差异点即高价值假设来源。两者都只为缩小假设空间、喂给下面的假设清单，不替代红信号闭环——闭环仍是 fail-closed 前提，没闭环就 diff 出方向也不算解决。

## 假设纪律（诊断特化）

清单纪律（3-5 条可证伪、排序、写预测、一次只动一个变量、先给用户看一眼）以 `references/dao/inquiry-discipline.md`「假设纪律」为准；诊断里每条预测具体化为“若 X 是因，则改 Y 会让 bug 消失 / 改 Z 会更糟”，写不出这种预测的是直觉，丢掉或磨快。

## 探针与清理

优先 debugger / REPL 断点（一个断点胜十条日志），其次只在区分假设的边界打定向日志，禁止“全量打日志再 grep”。每条调试日志打唯一前缀（如 `[DEBUG-a4f2]`），收尾用一条 grep 清干净——不打前缀的会残留。性能回归靠日志通常是错的：先量基线（计时 harness、profiler、查询计划）再二分，先量后修。

## 收口

回到 `references/dao/review-kit.md` 诊断起草结构与 `implement-code` 收尾审查门收口；额外确认：原始（未最小化）场景已不再复现、回归测试已落位（确无接缝就把“无接缝”本身记为发现）、带前缀探针已 grep 清净、最终成立的那条假设写进 commit / PR 供下一个人复用。
