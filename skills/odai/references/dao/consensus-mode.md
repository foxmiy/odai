# 合议增强模式

## 何时读取

仅在用户显式激活 `增强模式` / `合议` / `多模型复核` / `开合议`，或按 `references/dao/decision-challenge.md` 征询后用户同意时读取。默认路径不加载。本模式前提是宿主真实暴露多模型或可下放 agent；不得伪造。

## 席位与模型

- 默认 3 席；用户指定席位数和模型时从用户。
- 至少 2 种异构模型，最多 1 席与主流程同模；不得用同模型副本、换 agent 名或换提示词冒充异构。
- 点名模型不可用时，先剔除或回问；不得先宣称已合议再把失败藏到结尾。
- 模型倾向只作经验偏好，必须有当前宿主证据：规划 / 谋定可偏 Claude；设计 / 视觉可偏 Gemini；复核 / 找漏可偏 GPT 或 GLM；大体量低风险任务仅在宿主已证实低成本档满足要求时下放。用户点名高于本表。

## 三类用法

1. **重大决策 / 方案合议**：题面锁定、结论未冻结时，N 席围绕同一题面各自独立找反例、漏判、备路和不可接受结果；主流程收束，合议不替代用户确认。
2. **待冻结方案复查**：主路 / 边界 / 验收已形成待冻结结论时，至少 1 席独立挑战，主流程采纳 / 部分采纳 / 驳回后才能冻结。
3. **执行分工**：方案冻结后，把可切分子任务下放给有当前证据支持的匹配模型；单席只承接冻结题面、边界与验收，不得擅改路线。

## 下发包

每席一包：冻结题面 / 子任务范围、本席位模型与选择理由、关注点 / 切入点、同版 `odai` 契约与 `references/dao/agent-governance.md` 边界、输出格式与收口要求。

## 收口

- 结论冲突时不按多数票，按证据强弱和是否动摇道 / 术 / 边界裁决；足以动摇上游的，回写重定。
- 收口写明采纳了哪些席位的哪些点、驳回了什么、为什么。
- 宿主只给同系模型或给不全点名模型时，问用户按当前可用最高档继续，还是退回自动 1 席挑战；无 agent 能力则主流程内显式反证。
- 结果总结必须如实回报实际席位数、模型、分工、采纳 / 驳回结论；未真实派起的模型不得列入。
