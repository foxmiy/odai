# Planning Playbook

## 何时读取

由 `feature-plan` 触发：附件 / 低信息输入、分层步进提问、多方案比较、结构化对象属性完备性或规划文档草案。

## 最小顺序

1. 写当前理解；拿不准即列未确认点。
2. 视需要用九层透镜排查缺口：真正目标、成功标准、输入 / 触发、输出 / 结果、范围 / 非目标、表层诉求、用户手段、约束声称、已有证据；已清楚的层不必机械展开。
3. 输入少或乱时，先整理已知、未知、冲突点和必须确认的问题，不只回“请补充”。
4. 需要文档时先写最小草案；位置按当前对话或项目 Markdown 选择。

## 低信息与理解扩展

- 有附件 / 参考材料时，先分析目标、显性规则、隐性前提、遗漏、与项目现状冲突，以及哪些可采用、哪些须确认。
- 几乎没需求时，先查项目说明、相似功能、已有模式和约束；无足够参照时先问缺失前提。
- 表层诉求不等于真正目标；用户手段不等于最优路线；约束声称不等于硬约束。
- 若用户已指定方案，反推核心问题，并判断是否有更小、更稳、更省长期成本的路线。
- 主动补看相邻场景、边界条件、失败路径、上下游依赖、环境差异、流程误用、维护与协作成本。
- 结构化对象或参数集要推敲生命周期、状态、关联、检索分类、追溯合规；候选字段只作待确认提议。

## 提问、草案与比较

- 分层提问：目标层、场景层、边界层、落地层；同层一次问完，只问当前最影响治理判断的层。可由材料补齐的层先补证，不问用户。
- 探索期草案：当前理解、待确认问题、未确认路径、本轮建议下一步。
- 完整方案按需补：目标与非目标、约束与风险、方案选项、执行拆解、用户文档、AI 文档。
- 有冲突或多解释路径时，必须写出冲突和待确认解释。
- 多方案比较给真正不同的 2-3 个方案，至少含最小可行与更彻底对照；纳入不做 / 观察、复用现有能力、绕过问题、退回上游。
- 用同一维度比较：解决程度、改动面、风险、维护成本、一致性、可逆性、依赖与前提。
- 推荐项说明为何匹配用户权衡、代价是什么、何时应改选；未经确认不得当既定路径。并行辅助按 `references/dao/agent-governance.md`。
